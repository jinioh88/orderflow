# 04. 앱 컴포넌트 스펙 — 「5분 현장 도구」

> 소유: PM/디자이너 세션. 선행: [01-foundations.md](01-foundations.md) · [02-patterns.md](02-patterns.md)
> 시각 레퍼런스: [styleguide-app.html](styleguide-app.html) (브라우저로 열어 대조 — 폰 프레임 목업)
> 값은 토큰 이름으로 기술 — hex는 [tokens.json](tokens.json). 단위는 dp.

박점주의 사용 맥락 2가지가 모든 결정의 기준이다:
**① 영업 마감 후 피곤한 상태에서 5분 발주** (큰 글자·큰 타깃·최소 뎁스) /
**② 창고에서 한 손으로 바코드 검수** (다크 고정·물리 피드백·오프라인 내성).

## 1. 앱 셸

### 1.1 하단 탭 (4탭 — D0 결정 #5)

| 탭 | 아이콘(Material) | 내용 |
|------|------|------|
| 홈 | `home` | 마감 카운트다운, 오늘 발주 요약, 템플릿 바로가기, 진행 중 발주 상태, **오늘 도착 예정 카드**(M4) |
| 발주 | `list_alt` | 카탈로그 탐색 → 장바구니 → 제출 (템플릿 진입 포함) |
| 내역 | `receipt_long` | 발주 목록·상태·상세 타임라인 (US-ORD-05) |
| 더보기 | `menu` | 입고 확인(M4), 재고(v1.1), 알림 설정, 계정 |

- 높이 56 + 세이프에어리어. `surface` 배경 + 상단 `border`. 활성: `app.primary` 아이콘+라벨(12/600),
  비활성: `textCaption`. 뱃지: 내역 탭에 진행 중 상태 변경 수(danger dot)는 v1.1(NTF-02)에서.
- **입고 확인 주 진입로는 홈이다** (2026-07-19 외부 리뷰 반영): M4에서 출하 확정(SHIPPED) 건이
  있으면 홈 상단(마감 배너 다음)에 **'오늘 도착 예정' 카드** — 출하 건 요약 +
  `[바코드 검수 시작]` 버튼으로 스캔 화면 1-Click 진입. 더보기 탭의 메뉴는 보조 진입로.
  독립 탭 승격은 M4에서 사용 빈도 확인 후 재검토 (기존 방침 유지).

### 1.2 화면 구조

- 앱바: 높이 56, 화면 타이틀 app.title(22). 뒤로가기 외 액션은 우측 최대 2개.
- 콘텐츠 좌우 여백 16, 섹션 간격 24, 카드 간격 12.
- **하단 고정 CTA 바**: 주 액션이 있는 화면(장바구니, 검수 등)은 하단 고정 —
  `surface` + 상단 border + shadow-2, 패딩 12/16, 버튼 1개(전폭) 또는 요약+버튼.

## 2. 버튼 · 입력 (앱 변형)

| 변형 | 스펙 | 용도 |
|------|------|------|
| CTA | h=52, 전폭, `app.primary` 배경 + 흰 텍스트 app.body-strong(16/600), radius-md(12) | 하단 고정 주 액션 (발주 제출, 입고 확정) |
| 기본 | h=48, `app.primary` 또는 secondary(`surface`+`border`) | 카드 안 액션 (템플릿 불러오기) |
| 텍스트 | h=48(터치 영역), `app.primaryStrong` 텍스트 | 보조 (전체 보기, 취소) |
| danger | 기본과 동일 몸체, `feedback.danger.solid` | 발주 취소 — 확인 다이얼로그 세트 |

- 터치 타깃 48 미만 금지, 인접 간격 8 이상 (파운데이션 §6). 로딩·disabled 규칙은 웹과 동일(D2·03 §2).
- **오렌지+흰 텍스트 조합은 버튼 위젯 내부에서만 허용** (2026-07-19 외부 리뷰 반영):
  대비 3.2:1 제약(파운데이션 §1.2)을 사람이 기억하게 두지 않는다 — CTA/기본 버튼 위젯이
  이 조합을 캡슐화하고, 화면 코드에서 `app.primary` 배경 + 흰 텍스트를 직접 조합하는 것은 금지.
  본문 크기 오렌지 텍스트는 `primaryStrong` 상수만 노출해 구조적으로 강제한다.
- **인풋**: h=48, radius-sm(8), 텍스트 app.body(16). 포커스 보더 `app.primary`.
  수량·금액은 우측 정렬 + tabular + 단위 서픽스. 검증 규칙은 D2 §3.
- **수량 스테퍼**: `[−] 12 [+]` — 버튼 44×44(간격 포함 터치 48), 값 app.num-xl(24/700 tabular) 중앙,
  값 탭 시 숫자 키패드 직접 입력. 0에서 −는 장바구니 제거(확인 없이, 토스트+실행취소 — D2 §4.4).
  길게 누르면 연속 증감(200ms 간격).

## 3. 발주 플로우 컴포넌트

### 3.1 카탈로그 탐색 (US-ORD-01)

- **검색 바**: h=44, 상단 고정, `surfaceAlt` 배경 필드형. 최근 검색어 3개 표시.
- **카테고리 칩**: h=36, 가로 스크롤 1행, radius-full. 활성 = `primaryBg` 배경 + `primaryStrong`
  텍스트 + 보더. "전체"가 항상 첫 번째.
- **상품 행 카드**: 리스트형(그리드 아님 — 품명 가독 우선). 높이 최소 72.
  구성: 품명 app.body-strong(1줄 말줄임) / 단가 + 발주 단위 caption (`₩12,000 / 박스`) /
  우측에 수량 스테퍼 또는 [담기] 버튼(수량 0일 때). 판매중지 품목은 목록에서 제외(회색 표시 아님).
  한정 품목은 품명 옆 `한정` 미니 뱃지(warning 계열) + 잔여 수량 표시.

### 3.2 장바구니 요약 바 & 제출

- 카탈로그 화면 하단 고정: `12품목 · ₩1,240,000` (tabular) + [장바구니 보기] → 장바구니 화면.
- 장바구니: 상품 행 카드 재사용(스테퍼 포함) + 스와이프 삭제. 하단 CTA `발주 제출 — ₩1,240,000`.
- 제출 화면 상단에 **마감 상태 배너 상시** (§4.2) — 오늘 마감분인지 이월분인지 제출 전에 반드시 인지.
- 제출 성공 화면: success 아이콘 + `발주 완료` + **예상 출하일**(cutoff 회차 기반) + [내역 보기].
  멱등키는 UI 개념 아님 — 중복 제출 시에도 같은 완료 화면(NFR-3.3, 서버가 기존 발주 반환).

### 3.3 발주 템플릿 (US-ORD-02 — 1-Click)

- 홈 상단 + 발주 탭 진입부에 노출. **카드형 가로 스크롤** (w=200):
  템플릿 이름 app.heading / `n품목 · 약 ₩000,000` / [불러오기] 기본 버튼.
- 불러오기 = 장바구니에 즉시 적재 후 장바구니 화면으로 이동 (수량 조정 → 제출. 목표: 탭 3번).
  기존 장바구니가 있으면 병합/교체 선택 다이얼로그.
- 템플릿 저장: 장바구니·발주 완료 화면에서 "템플릿으로 저장" 텍스트 버튼.

## 4. 상태 표현

### 4.1 발주 카드 (내역 목록) & 상태 타임라인 (US-ORD-05)

- **발주 카드**: radius-lg(16), `surface`, border, 패딩 16.
  1행: badge-lg + 발주번호 caption / 2행: `7품목 · ₩1,240,000` app.body-strong /
  3행: 제출 시각 + 예상 출하일 caption. REJECTED는 사유 1줄 danger.text로 카드에 노출(D2 §1.2).
- **타임라인** (상세 화면): 세로형, 해당 발주의 경로만(D2 §1.2). 완료 단계 = 상태색 dot(12) + 시각,
  현재 단계 = dot 16 + 상태색 라벨 강조, 미래 단계 = `neutral-300` dot + 라벨.
  거절/취소로 종결된 발주는 종결 노드가 마지막(미래 단계 없음).

### 4.2 마감 배너 (US-ORD-04 · PD-2) — 홈·장바구니·제출 화면 공통

| 상태 | 색 | 문구 |
|------|------|------|
| 마감 60분 전~ | warning | `발주 마감까지 29:59` — mm:ss 카운트다운(파운데이션 §5) |
| 60분 이상 남음 | 배너 없음 (홈 요약 카드에 `오늘 12:00 마감`만) | |
| 마감 후 | info | `지금 제출하면 익영업일(7월 20일) 마감분으로 접수됩니다` |

- 이월은 **info** — 오류처럼 보이면 안 됨(D2 §4.5). 카운트다운 00:00 도달 시 즉시 info 배너로 전환.

## 5. 스캔 검수 화면 (US-GRN-01~02) — 다크 고정

D0 결정 #4의 유일한 다크 화면. `scanScreenBg`(neutral-900) 배경 + `scanScreenText` 텍스트.

```
┌──────────────────────┐
│ ← 입고 검수  출하건 SI-0412 │  앱바 (다크)
│   ┌─ 뷰파인더 ─┐        │  카메라 프리뷰, 중앙 스캔 프레임(모서리 4개 브래킷)
│   └────────────┘        │  인식 순간: 프레임 success 색 플래시 + 진동 + 비프
│ ▒ 방금: 콜드브루 원액 1L    │  인식 결과 시트(하단): 품명 + 검수 수량 자동 +1
│   검수 8 / 출하 10  [+1][수정] │  연속 스캔 대응 — 시트는 갱신만, 화면 전환 없음
│ [ 목록 보기 ]  [ 검수 완료 ] │
└──────────────────────┘
```

- **인식 피드백 3중**(파운데이션 §6): 프레임 플래시(success) + 진동 + 소리. 식별 0.5초 이내(NFR-1.4).
  동일 바코드 연속 인식 시 +1 누적(쿨다운 1초 — 같은 실물 중복 카운트 방지), 카운트 배지가 튀는 마이크로 애니메이션.
- **미등록 바코드**: 프레임 danger 플래시 + 다른 진동 패턴 + 시트에 `이 출하 건에 없는 품목입니다` (스캔은 계속 가능).
- **목록 보기**: 바텀 시트(밝은 surface) — 품목별 `검수 n / 출하 m` 리스트. 완료 라인 success 체크,
  부족 라인은 수량 warning.text. 수동 조정(스테퍼) 가능 — 바코드 훼손 대비.
- **검수 완료 CTA**: 전 라인 일치 시 활성 문구 `입고 확정`. 불일치 존재 시 `검수 완료 — 불일치 2건` →
  클레임 플로우(§6)로 연결. 확정은 1회만·불가역이므로 확인 다이얼로그(D2 §4.4 — 결과 요약 포함).
- 오프라인에서도 검수 지속(로컬 반영 — US-GRN-01 인수 조건). 오프라인 배너는 다크 화면에서는 상단 얇은 라인으로.

## 6. 클레임 등록 폼 (US-GRN-03 · PD-3)

- 진입: 검수 완료 시 불일치 라인이 **자동으로 클레임 초안**이 됨(불일치 수량 = 출하 − 검수, 수정 불가 —
  도메인 불변식). 점주는 유형·사진만 보탠다.
- 라인별 카드: 품명 + `출하 10 · 검수 8 · 불일치 2` (tabular) /
  **유형 세그먼트**: `파손` `누락` (h=44, 단일 선택 필수) /
  **사진 첨부**: 카메라 즉시 촬영 우선(갤러리 보조), 썸네일 80×80 가로 나열, 최대 5장. 파손은 사진 권장 문구.
- **안심 고지는 폼 최상단에 info 박스로** (2026-07-19 외부 리뷰 반영 — 격상):
  `불일치분은 청구되지 않습니다 — 정산은 실제 입고 수량 기준입니다` 를 유형 선택보다 먼저,
  info 배너 박스(D2 §4.5 스타일)로 상시 노출한다. 클레임은 점주에게 심리적 부담이 있는
  행위("내 귀책인가, 추가 비용인가")이므로, 안심 정보가 입력 행위보다 앞서야 한다.
- 하단 CTA `클레임 접수 · 입고 확정` — 클레임 접수와 부분 입고 확정이 한 액션임을 라벨로 명시.

## 7. 오프라인 (US-ORD-07 — M5, 스펙 선행)

- **오프라인 배너**: 전 화면 상단 고정, `neutral-800` 배경 + 흰 텍스트 —
  `오프라인 — 발주는 저장 후 연결되면 자동 전송됩니다`. 에러 색을 쓰지 않는다(모드이지 오류가 아님, D2 §2.3).
- **전송 대기 발주 카드**: 상태 뱃지 자리에 `전송 대기` 회색 필 + `schedule_send` 아이콘
  (서버 상태 아님 — 7상태 뱃지와 시각적으로 구분: dot 대신 아이콘). 카드에 저장 시각 표기.
- 연결 복구: FIFO 자동 전송(UI 개입 없음) → **동기화 안심 배너** (2026-07-19 외부 리뷰 반영 —
  토스트에서 격상): 홈·내역 상단에 success 배너
  `네트워크 복구 — 보류 중이던 발주 2건이 전송되었습니다 (12:05)` 를 노출한다.
  토스트는 사라지면 끝이라 "정말 전송됐나"의 불안을 못 잡는다 — 배너는 점주가 닫을 때까지
  유지(X 버튼), 탭하면 해당 발주 내역으로 이동. 전송 시각 명시 필수.
  전송 시점이 마감을 지나 이월된 경우 해당 카드에 info 라인 + 푸시(PD-2 고지).
- 제출 버튼은 오프라인에서도 동일 동작(로컬 저장) — 비활성화하지 않는다.
