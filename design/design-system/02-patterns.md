# 02. 공통 패턴

> 소유: PM/디자이너 세션. 선행: [01-foundations.md](01-foundations.md) — 색·타이포 값은 전부
> [tokens.json](tokens.json)의 토큰 이름으로 기술한다.
> 여기 정의된 패턴은 웹·앱 공통 규칙이며, 플랫폼별 형태 차이는 각 절의 Web/App 변형으로 명시한다.

## 1. 상태 뱃지 (Order Status Badge)

발주 상태 7종(`semantic.color.orderStatus.*`)의 시각 표현. **라벨 텍스트는 토큰의 `label` 값을
그대로 쓴다** — 화면마다 다른 표현("승인완료", "완료됨" 등) 금지.

### 1.1 형태 스펙

| 변형 | 플랫폼·용도 | 구성 | 크기 |
|------|------|------|------|
| badge-sm | 웹 그리드 셀 | 필 배경(`bg`) + 라벨(`text`) | 높이 20, 패딩 2×8, radius-full, web.caption(12) 500 |
| badge-md | 웹 상세·모달, 앱 목록 행 | dot(6px) + 라벨 | 높이 24, 패딩 4×10, radius-full, 웹 13/앱 14, 500 |
| badge-lg | 앱 카드 헤더 | dot(8dp) + 라벨 | 높이 32, 패딩 6×12, radius-full, app.body-strong(16) |

- dot 색은 `dot`, 라벨 색은 `text`, 배경은 `bg` 토큰. 보더 없음.
- 그리드 정렬을 위해 badge-sm의 라벨은 줄바꿈 금지(최장 "승인 대기" 4자 기준 고정 폭 아님, 내용 폭).
- 상태 전이 애니메이션(낙관적 업데이트 등)은 배경색 200ms ease 트랜지션만. 움직임 효과 금지.

### 1.2 상태 표현 보조 규칙

- **REJECTED는 항상 사유 접근 수단을 동반한다** — 웹: 셀 호버 툴팁 + 상세 패널, 앱: 카드에 사유 1줄 노출 (US-RCV-01 사유 필수의 대응).
- **PD-2 이월 발주**: SUBMITTED 뱃지 옆에 보조 텍스트 `익영업일 마감분 (M월 D일 출하 예정)` — warning 색이 아닌 caption 색. 이월은 오류가 아니다.
- 앱 발주 상세의 **상태 타임라인**(US-ORD-05)은 7상태 전체가 아니라 **해당 발주가 지나온/지나갈 경로만** 표시 (예: 제출됨 → 승인됨 → 출하됨 → 입고 완료). 미래 단계는 neutral-300.

## 2. 빈 상태 · 로딩 · 에러

### 2.1 빈 상태 (Empty State)

구성: 아이콘(도메인 매핑 표의 해당 개념, neutral-300, 웹 32/앱 48) + 제목 1줄 + 설명 1줄 + (가능하면) 액션 버튼.

| 상황 | 제목 톤 | 액션 |
|------|------|------|
| 아직 데이터 없음 (첫 사용) | "등록된 상품이 없습니다" | 만들기 CTA ("상품 등록") |
| 필터 결과 없음 | "조건에 맞는 수주가 없습니다" | "필터 초기화" |
| 오늘 할 일 없음 (앱 홈 등) | 긍정적으로 — "오늘 발주가 완료되었습니다" | 없음 또는 보조 링크 |

- 필터 결과 없음을 "데이터 없음"처럼 보여주지 않는다 — 사용자가 필터를 의심하게 만들어야 한다.

### 2.2 로딩

| 상황 | 패턴 |
|------|------|
| 목록·카드 최초 로드 | **스켈레톤** (neutral-100 바탕, 1.5s 셔머). 실제 레이아웃과 동일한 형상 — 그리드는 행 스켈레톤, 카드는 카드 스켈레톤 |
| 버튼 액션 진행 | 버튼 내 스피너(16px) + 라벨 유지 + disabled. 버튼 폭 변화 금지 |
| 재조회(이미 데이터 있음) | 기존 데이터 유지 + 우상단 미세 스피너. **화면 전체를 로딩으로 덮지 않는다** |
| 300ms 미만 응답 | 로딩 표시 안 함 (깜빡임 방지 — 300ms 지연 후 표시) |

### 2.3 에러

| 상황 | 패턴 |
|------|------|
| 화면 단위 실패 (목록 로드 실패) | 빈 상태와 동일 구성 + danger 아이콘 + **"다시 시도" 버튼 필수** |
| 부분 실패 (한 카드만) | 해당 영역만 인라인 에러 + 재시도. 화면 전체를 죽이지 않는다 |
| 네트워크 오프라인 (앱) | §5.4 오프라인 배너 — 에러가 아닌 **모드**로 취급 (US-ORD-07) |
| 권한 없음/404 | "찾을 수 없습니다" 단일 화면 — 테넌트 격리(US-AUTH-04)에 의한 404와 실제 404를 구분해 보여주지 않는다 |

## 3. 폼 패턴

- **라벨은 인풋 위** (플로팅 라벨 금지 — 40대 가독성·스캔 용이). 웹 web.body-md 500, 앱 app.caption 500.
- **필수 표기**: 필수 항목이 과반이면 선택 항목에 `(선택)`을 붙이고 필수 표기는 생략. 그 반대면 필수에 `*`.
- 단위가 있는 입력(수량·금액·한도)은 인풋 우측에 단위 서픽스를 인풋 안에 고정 표시.
- **검증 시점**: 블러 시 1차, 제출 시 전체. 타이핑 중 실시간 에러 금지 (입력이 끝나기 전에 혼내지 않는다). 에러 상태에서 재입력 시작하면 즉시 에러 해제.
- 제출 실패 시 첫 에러 필드로 스크롤+포커스.

### 3.1 검증 메시지 톤 — "원인 + 해결"

| ❌ | ✅ |
|------|------|
| 잘못된 입력입니다 | 수량은 1 이상이어야 합니다 |
| 형식 오류 | 이메일 형식이 아닙니다 (예: name@company.com) |
| 저장 실패 | 품목코드 'A-101'이 이미 존재합니다. 다른 코드를 입력하세요 |

- 메시지는 danger.text 색, 인풋 아래 12/14px, 아이콘 없이 텍스트만.
- 서버 검증 실패(400)의 필드 오류는 해당 필드 아래에, 전역 오류는 폼 상단 배너에.

## 4. 피드백 패턴

### 4.1 토스트

| 속성 | Web | App |
|------|------|------|
| 위치 | 우하단, 위로 쌓임 (최대 3개) | 하단 CTA 위, 1개만 (교체) |
| 지속 | 성공 3s / 오류 5s | 성공 2.5s / 오류 4s |
| 구성 | 아이콘 + 메시지 + (선택) 액션 버튼 1개 + 닫기 | 아이콘 + 메시지 + (선택) 액션 |
| 색 | surface 배경 + 좌측 4px 시맨틱 컬러 바 | 동일 |

- 토스트로 **중요 정보를 전달하지 않는다** — 사라져도 업무가 되는 보조 확인용. 판단이 필요한 내용은 다이얼로그나 배너.

### 4.2 낙관적 업데이트 & 롤백 (US-RCV-03 — 웹 수주 승인/거절의 핵심)

```
클릭 → 즉시 그리드 셀 뱃지를 새 상태로 전환 (200ms 트랜지션)
     → 성공: 그대로. 토스트 없음 (뱃지 변화가 곧 확인)
     → 실패: 뱃지를 원래 상태로 롤백 + danger 토스트
             "승인 실패 — 이미 김대리 님이 거절한 건입니다" (서버 사유 그대로 노출)
```

- 롤백 토스트에는 **누가/왜 실패했는지 서버가 준 사유를 그대로** 담는다 (동시 처리 충돌 안내).
- 처리 중(pending) 시각 표시는 하지 않는다 — 낙관적 UI의 목적은 즉답감. 단 롤백은 반드시 시각적으로 원위치.
- 일괄 승인(US-RCV-04, v1.1)은 "n건 성공 · m건 실패" 요약 토스트 + 실패 건 필터 액션.

### 4.3 진행률 (US-CAT-02 — 엑셀 업로드)

- 0~100% **결정형 프로그레스 바** (primary 색) + "n / 1,000행 검증 중" 카운트 텍스트.
- 완료 후 두 갈래:
  - 전체 성공 → success 요약 ("1,000행 등록 완료")
  - 오류 존재 → **전체 거부** 결과 화면: danger 요약 ("업로드 거부 — 12행에서 오류") + 행 번호·오류 사유 테이블 + "오류 리포트 다운로드". 정상 행이 몇 건인지도 표기해 전체 거부 정책(인수 조건)임을 명시.
- 진행 중 모달 닫기 금지(업로드 취소 버튼으로만 이탈).

### 4.4 확인 다이얼로그

| 등급 | 대상 | 형태 |
|------|------|------|
| 확인 | 상태 전이 (출하 확정, 입고 확정 — 이후 불가역) | 제목 + 결과 요약("발주 12건이 SHIPPED로 전이됩니다") + 확인(primary)/취소 |
| 파괴 | 거절, 발주 취소, 계정 비활성화 | 동일 + 확인 버튼 danger 색. 거절은 사유 입력 필드 포함(필수) |

- 버튼 라벨은 동사로 — "확인/취소"가 아니라 **"출하 확정" / "거절"**. 취소만 "취소".
- 되돌릴 수 있는 액션(장바구니 삭제 등)에는 다이얼로그 대신 토스트+실행취소를 쓴다.

### 4.5 배너 (지속 상태 고지)

토스트가 "사건"이면 배너는 "상태"다. 상태가 해소될 때까지 화면 상단에 유지.

| 배너 | 플랫폼 | 색 | 내용 |
|------|------|------|------|
| 마감 임박 | App | warning | "발주 마감까지 29:59" — 60분 전부터, 카운트다운(mm:ss) |
| 마감 후 이월 안내 (PD-2) | App | info | "지금 제출하면 익영업일(7월 20일) 마감분으로 접수됩니다" — 장바구니·제출 화면 |
| 오프라인 (US-ORD-07) | App | neutral-800 배경 + 흰 텍스트 | "오프라인 — 발주는 저장 후 연결되면 자동 전송됩니다" + 전송 대기 n건 |
| 배치 실패 등 시스템 이슈 | Web | danger | 관리자 고지 (M5+) |

- 이월 안내는 **info**다 — warning으로 쓰면 정상 정책(PD-2)이 오류처럼 읽힌다.

## 5. 마이크로카피 원칙

1. 용어는 유비쿼터스 언어(01-overview 6장) 그대로 — 화면에서 "주문"이라 쓰지 않고 웹은 "수주", 앱은 "발주".
2. 높임 평서형, 명사형 종결 최소화 — "저장되었습니다" (O), "저장 완료됨" (X). 버튼은 동사 명령형("발주 제출").
3. 숫자를 앞세운다 — "승인 대기 7건"이 "승인 대기 건이 7개 있습니다"보다 낫다 (특히 웹 관제탑).
4. 앱은 시간 압박을 존중 — 한 화면 안내문은 1문장, 부가 설명은 접거나 생략.
5. 시스템 자신을 주어로 하지 않는다 — "시스템이 발주를 거부했습니다" (X), "잔여 수량이 부족합니다 (현재 3개)" (O — US-ORD-06 응답 형식).
