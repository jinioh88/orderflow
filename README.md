# 01. 개요 & 기획 배경

> OrderFlow — B2B 프랜차이즈 가맹점 수발주 & 물류 관리 플랫폼

## 1. 해결하려는 문제

프랜차이즈 본사와 가맹점 간 식자재 수발주는 여전히 **카톡·엑셀·전화**로 이루어지는 경우가 많다.
이 방식은 다음과 같은 구조적 문제를 만든다.

| 문제 | 원인 | 결과 |
|------|------|------|
| 오배송·수주 누락 | 발주 채널이 분산되어 있고 수기 취합 | 가맹점 영업 차질, 본사 신뢰도 하락 |
| 정산 지연 | 가맹점별 미수금을 엑셀로 수기 관리 | 월 마감에 3~4일 소요 |
| 발주 시간 부담 | 복잡한 웹 발주, 마감 시간 인지 어려움 | 점주가 마감을 놓쳐 재고 결품 발생 |
| 재고 불투명 | 입고 확인이 수기 카운트에 의존 | 실재고와 장부 재고의 오차 누적 |

OrderFlow는 **발주 접수 → 정시 마감 집계 → 출하 지시 → 바코드 입고 확인 → 정산/미수금**까지의
흐름을 하나의 플랫폼으로 통합하여 이 문제를 해결한다.

## 2. 타깃 페르소나

### 👤 김운영 — 프랜차이즈 본사 물류/운영 팀장 (30대 후반, **Web**)

- **목표**: 전국 가맹점의 발주 내역을 한눈에 파악하고, 물류 센터로 출하 지시를 정확하고 빠르게 전달
- **Pain Point**
  - 카톡·엑셀·전화로 제각각 들어오는 발주 때문에 오배송/누락이 잦음
  - 가맹점별 미수금 관리와 월간 매출 집계에 매달 3~4일 소요
- **요구사항**
  - 매일 정해진 마감 시간에 발주 데이터가 자동 집계
  - 엑셀 대량 업/다운로드, 가맹점별 미수금 현황 대시보드

### 👤 박점주 — 가맹점(카페/음식점) 점주 (40대 초반, **Mobile**)

- **목표**: 매일 영업 마감 후 5분 이내에 식자재를 정확하게 주문하고 입고를 확인
- **Pain Point**
  - 복잡한 웹 발주가 번거롭고, 마감 시간을 놓쳐 재고 결품 발생
  - 창고 원자재를 매번 직접 수량 카운트해야 함
- **요구사항**
  - 모바일 앱에서 장소 제약 없는 빠른 1-Click 발주
  - 바코드 스캔 간편 입고 확인, 발주 마감 전 푸시 알림

## 3. 프로젝트 목표

### 3.1 비즈니스 / 사용자 목표

| 목표 | 측정 지표 |
|------|-----------|
| 발주 처리 시간 단축 | 전화/엑셀 대비 **80% 이상 단축** (발주 완료까지 5분 이내) |
| 오배송·누락 Zero화 | 자동 집계 시스템으로 **인적 실수로 인한 수주 누락률 0%** |
| 재고·미수금 투명성 | 실시간 정산 + 바코드 입고 확인으로 **재고 오차율 최소화** |

### 3.2 기술적 목표 

**Backend & DB**
- **멀티테넌트 격리**: Tenant-ID 기반 데이터 접근 통제 + Hibernate Filter 구조적 강제로 **데이터 누출 0건** — 테넌트 교차 접근 시나리오 자동화 통합 테스트 100% 통과로 검증
- **대용량 정시 집계**: 매일 정오 마감 시 Spring Batch(Chunk 기반)로 **발주서 10,000건 / 발주 품목(Line Item) 50,000건** 기준 3초 이내 집계
- **재고 동시성 제어**: 한정 재고 품목 동시 주문 시 Redis Redlock으로 Overbooking 방지

**Frontend (React)**
- **대용량 그리드 최적화**: AG-Grid + Virtual Scroll로 1,000건 이상 목록 렌더링 지연 제거
- **낙관적 업데이트**: React Query로 수주 승인/거절 시 즉각적인 UI 반응성

**Mobile (Flutter)**
- **현장 친화 UX**: mobile_scanner로 0.5초 이내 바코드 인식 및 로컬 재고 반영
- **오프라인 대응**: 네트워크 불안정 시 발주 데이터를 Isar에 저장, 복구 시 제출 순서를 보장하며 자동 재전송 (멱등키로 중복 방지)

**DevOps & Monitoring**
- **무중단 배포**: GitHub Actions + Docker + AWS EC2/S3/CloudFront 완전 자동화 파이프라인
- **관제 체계**: Prometheus + Grafana로 발주 마감 피크 시간대 JVM Heap / DB 커넥션 풀 실시간 모니터링

## 5. 핵심 사용자 흐름 (Happy Path)

```
[박점주 · Mobile]                [System · Batch]              [김운영 · Web]
     │                                 │                             │
 1. 영업 마감 후 앱에서                  │                             │
    자주 쓰는 품목 1-Click 발주 ────────▶│                             │
     │                                 │                             │
 2. (D+1 11:30) 마감 30분 전            │                             │
    미발주 가맹점 푸시 알림 ◀────────────│                             │
     │                                 │                             │
     │                          3. (D+1 12:00) 발주 마감              │
     │                             Batch가 당일 발주 집계,             │
     │                             출하 지시서 자동 생성 ──────────────▶│
     │                                 │                             │
     │                                 │                      4. 수주 현황 확인,
     │                                 │                         예외 건 승인/거절,
     │                                 │                         출하 확정
     │                                 │                             │
 5. 물품 도착 → 바코드 스캔으로          │                             │
    입고 확인, 가맹점 재고 자동 반영      │                             │
     │                                 │                             │
     │                          6. 입고 확정 건 정산 반영,             │
     │                             미수금 대시보드 갱신 ───────────────▶│
```

## 6. 용어 정의 (Ubiquitous Language)

| 용어 | 영문 | 정의 |
|------|------|------|
| 테넌트 | Tenant | 플랫폼을 사용하는 프랜차이즈 본사. 데이터 격리의 단위 |
| 가맹점 | Store | 테넌트에 소속된 발주 주체 (카페/음식점 지점) |
| 발주 | Purchase Order | 가맹점이 본사에 물품을 주문하는 행위 (가맹점 관점) |
| 수주 | (Received) Order | 본사가 가맹점의 발주를 접수한 것 (본사 관점, 동일 데이터) |
| 발주 마감 | Order Cutoff | 당일 출하 집계에 포함되는 발주의 접수 기한 (기본 매일 12:00) |
| 집계 | Aggregation | 마감 시점에 발주들을 가맹점/품목 단위로 묶어 출하 지시서를 만드는 배치 처리 |
| 출하 지시 | Shipment Instruction | 물류 센터에 전달되는 가맹점별 피킹/배송 지시서 |
| 입고 확인 | Goods Receipt | 가맹점이 도착 물품을 바코드 스캔으로 검수·확정하는 행위 |
| 한정 품목 | Limited-Stock Item | 본사 보유 재고가 유한하여 선착순 배분되는 품목 (동시성 제어 대상) |
| 미수금 | Receivable | 입고 확정되었으나 아직 수금되지 않은 가맹점별 채권 잔액 |
| 멱등키 | Idempotency Key | 클라이언트가 발주 제출마다 생성하는 유일 키. 네트워크 재전송·오프라인 재제출 시 동일 발주의 중복 생성을 방지 |
| 레드락 | Redlock | Redis 기반 분산 락 알고리즘. 다중 서버 환경에서 한정 품목 재고 차감의 동시성을 제어 |
